版本：2026-05-14
適用對象：私有版使用者、事務協作者、維運者

## 1. 私有版定位與使用原則

私有版 MAGI 是單一主機、單一事務環境的正式作業系統，目的不是展示功能，而是降低人工負擔並維持案件資料一致。

核心原則：

- 案件、法扶、行事曆、檔案與通知要以同一套資料為準。
- AI 可協助整理、摘要、搜尋、命名、草擬與提醒，但正式法律判斷仍由使用者確認。
- 同名不同案件不得混併。
- 批次搬檔、正式送出、刪除與 DB 還原要保留確認步驟。
- 公開版推送前必須通過脫敏與 public isolation。
- 私有 runtime、案件資料、NAS 路徑、OAuth token、金鑰與 DB dump 不得進 git。

若 MAGI 回報「需檢查」，先處理警示，再進行正式作業。

## 2. 每日啟動流程

每天使用前建議依序確認：

1. 主機已連網。
2. NAS 已掛載到正確名稱。
3. MAGI daemon 正常。
4. `/health` 顯示主狀態正常。
5. NERV 顯示資料庫、模型、OCR、向量資料庫、排程正常。
6. 通知頻道沒有未處理的嚴重錯誤。
7. 法扶與行事曆最近一次同步時間合理。
8. 磁碟空間沒有低水位。

命令：

```bash
cd ~/Desktop/MAGI_v2
./scripts/magi_cli.sh status
```

若異常：

```bash
./scripts/magi_cli.sh restart
```

重啟後先看健康狀態，不要立刻執行批次下載或批次搬檔。

## 3. 入口、健康狀態與外網

本機常用入口：

```text
http://127.0.0.1:5002/
http://127.0.0.1:5002/health
http://127.0.0.1:5002/dashboard/nerv
```

外網使用 Tailscale 或反向代理時，請以目前 NERV 或主機設定顯示的網址為準。若外網出現 `ERR_CONNECTION_CLOSED`，先檢查：

- MAGI daemon 是否正常。
- Tailscale 是否登入。
- Funnel 或反向代理是否開啟。
- 防火牆是否阻擋。
- `/health` 在本機是否正常。

可執行：

```bash
python3 scripts/ops/tailscale_funnel_healthcheck.py --apply
```

注意：`/openclaw` 已退役，預期回 404。請使用 MAGI 主頁、NERV 與 `/magi-adjust`。

## 4. OSC 案件管理

OSC 案件管理是私有版案件資料的核心。案件卡片與清單應至少能做到：

- 查看案件編號、當事人、案由、程序、法院案號、狀態。
- 開啟案件資料夾。
- 開啟、預覽、下載與分享檔案。
- 查看待辦、行事曆事件與 OSC 建立事件。
- 查看法扶資訊與回報狀態。
- 查看閱卷、律見、聯繫、會議、開庭、調解等統計。

同名不同程序時，判斷順序應為：

1. OSC 案件編號。
2. 法扶案號。
3. 法院案號。
4. 案由。
5. 程序。
6. 資料夾位置。
7. 當事人姓名。

不得只靠姓名判斷結案、搬移或統計。

範本案件是特殊資料，不應顯示一般案件狀態，也不應把 `upsert-smoke` 當作真實案由或案件分類。範本資料夾應在書狀頁籤下有獨立入口，可個別預覽、下載、分享與複製連結。

## 5. 法扶管理

法扶管理包含：

- 新派案建立。
- 官網附件下載。
- 開辦。
- 進度回報。
- 應備事項表。
- 消債文件確認。
- 結案。
- 已結案資料夾存取。

新派案流程：

1. 讀取派案通知。
2. 下載官網附件。
3. 若 email 資料不足，先建立臨時資料夾。
4. 讀取法扶申請書、案件概述單、審查表等附件。
5. 用附件內資訊修正當事人、案由、程序與資料夾名稱。
6. 若資料夾被開啟導致無法 rename，先關閉相關 Finder、Word、PDF 編輯器或同步程式，再重試。

開辦規則：

- 開辦是正式流程，不應出現「暫存」概念。
- 送出前仍需確認資料完整。
- 若 MAGI 顯示 go_live 暫存失敗，要檢查是否仍有舊流程殘留，修正後改走正式開辦流程。

消債應備事項：

- 應還原 OSC 邏輯。
- 所得清單以聲請前兩年為準。
- 民國 115 年 5 月後可取得 114 年資料，因此清單應為 113、114。
- 明年同期間應自動推進為 114、115。
- 待補資料要能輸出可複製文字。

## 6. 法扶結案與資料夾搬移

法扶結案流程：

1. 確認法扶官網狀態。
2. 確認案件是否已結案或可報結。
3. 檢查結案依據檔案。
4. 補齊必要附件。
5. 上傳或送出結案資料。
6. 標示已結案或已結案待送出。
7. 搬移資料夾到已結案區。
8. 確認原進行中路徑不再殘留同一案件資料夾。
9. 確認 MAGI 網頁版仍可開啟已結案資料夾與檔案。

結案判斷不得過寬：

- 同一當事人可能有更生、清算、強制執行、監護、刑事或民事不同程序。
- 結案只適用於特定法扶案號或 OSC 案件編號。
- 同名不同程序不可一起搬。

強制執行案件：

- 結尾可能是執行命令，不一定有債權憑證。
- 放在「判決書」資料夾內的執行命令，可作為結案依據候選。
- 若案件類型為清算，歷次程序裁定可能都要上傳。
- 若只需復權裁定與復權裁定確定證明，僅搬移與上傳這兩類檔案。

已結案存取：

- 已結案案件在 MAGI 網頁版仍要能開資料夾。
- 檔案預覽、下載、分享連結也要可用。
- 若資料夾移動後路徑失效，應更新 DB 與索引，不可要求使用者人工猜路徑。

## 7. 法扶活動計數與進度回報

法扶回報核心統計包含：

- 開庭。
- 調解。
- 會議。
- 電話聯繫。
- 律見。
- 閱卷。
- 文件補正與送出。

來源包含：

- OSC DB。
- Google Calendar。
- 法扶資料夾。
- 閱卷資料夾。
- 筆錄資料夾。
- 案件文件索引。

行事曆判斷：

- OSC 自動建立事件通常有案件系統編號。
- 法扶需計數事件，即使不是 OSC 自動建立，也要匯入統計。
- 一般手動私人行程不要混入案件待辦。
- 同名多案時，應用 DB 判斷是否為法扶案件，再用案號、案由與日期消歧。

閱卷次數：

- 以資料夾內有繳費單以外的資料為優先。
- 只有繳費單的日期資料夾不應計入閱卷完成。
- 舊閱卷模組若重複下載繳費單，不應讓統計膨脹。

進度回報：

- 進行中逾期案件應完整列出，不只顯示數字。
- 使用者可回覆「案號/姓名 已回報」或相似語意。
- MAGI 應將該案冷卻 60 天。
- 冷卻後應登上行事曆，提醒下一次回報。
- 進度回報提醒可發到 DC；一般法扶巡檢不應誤發到 DC 業務頻道。

## 8. 閱卷、筆錄與法院檔案

閱卷流程：

1. 檢查可下載案件。
2. 區分待繳費與可下載。
3. 下載後按案件歸檔。
4. 去重繳費單與重複檔案。
5. 更新閱卷統計。

筆錄流程：

1. 同步可下載筆錄。
2. 只在通知中列出本次有下載的案件。
3. 顯示下載檔名與案件。
4. 無下載案件不要占滿通知。

法院檔案命名：

- 優先使用 PDF 文字層。
- 沒有文字層才 OCR。
- 信封頁、封面頁、掃描邊界應先排除。
- 判決書資料夾、法院通知與程序裁定、對造書狀都要抽樣驗證。

## 9. PDF、OCR、命名與歸檔

PDF 命名目標：

- 法院通知要命名到日期、法院、案號、文件類型。
- 程序裁定要辨識裁定類型。
- 判決與確定證明要能抓出關鍵字。
- 對造書狀要能辨識書狀類型與日期。
- 掃描信封不應干擾命名。

命名流程：

1. 偵測頁面是否為信封或封面。
2. 使用文字層抽取。
3. 需要時啟用 OCR。
4. 套用法院文件規則。
5. 檢查日期是否合理。
6. 低信心進入人工確認。
7. 使用者修正後寫入同文件類型學習紀錄。

不可接受：

- 將民國年與西元年混淆。
- 把不合理日期直接命名。
- 將繳費單視為閱卷成果。
- 因為第一頁信封而錯命名整份文件。

## 10. 書狀產生、範本與學習回饋

書狀流程：

1. 從案件選擇書狀類型。
2. 選擇同案由或同程序範本。
3. 匯入案件事實、證據與既有書狀。
4. 產生 Word 草稿。
5. 保留原範本排版。
6. 輸出 PDF 檢查。
7. 與既有完稿抽樣比對。
8. 使用者修改後回饋學習。

學習回饋記錄原則：

- 記錄案由、程序、書狀類型、修改前、修改後、修改原因與時間。
- 僅同案由、同書狀類型優先套用。
- 不同案由不得混用。
- 不記錄不必要的個資到公開版。
- 完稿與草稿都要可回溯。

若書狀超過 10 頁，抽樣比對應優先檢查：

- 頁首、頁尾、頁碼。
- 字型、行距、段落。
- 表格與證據編號。
- 引用條文與裁判。
- PDF 輸出是否與 Word 版一致。

## 11. 行事曆、待辦與業務概覽

業務概覽應分成兩個區塊：

- OSC 建立待辦。
- 行事曆事件。

不要把兩者混在同一個待辦表格。原因是：

- OSC 建立待辦是案件規則產生。
- 行事曆事件可能來自 Google Calendar 或手動輸入。
- 法扶計數要納入必要行程，但一般私人行程不該混入案件待辦。

更新頻率：

- OSC 建立待辦每 6 小時更新。
- 行事曆事件每 6 小時更新。
- 大量過期事件可批次標示完成，但不得干擾法扶報結與進度回報。

介面要求：

- 寬版表格要讓日期、案號、類型、描述與操作不擠在一起。
- 有時間的待辦要保留完整時間。
- 描述過長時應換行，不應截斷重要資訊。
- 卡片可點開完整內容。

## 12. 帳務與固定支出

帳務匯入原則：

- 可從 Google Sheets 按月匯入。
- 排程可設週一與週五。
- 標識為非本人的項目不匯入。
- 固定支出與同事登錄項目要去重。
- 薪資、勞保、勞退、健保等固定支出不可重複計算。

固定支出修改後，應確認：

- 月報表金額正確。
- 同一月份不重複。
- 來源有標記。
- 可追溯匯入時間。

帳務是輔助管理，不應在未確認前作為報稅或正式會計唯一依據。

## 13. AI 問答、法律資料與 MCP

AI 問答應能依問題調用正確工具：

- 問行程：查行事曆與待辦。
- 問案件：查案件 DB 與資料夾。
- 問檔案：查檔案索引。
- 問法律資料：查知識庫與法律資料 MCP。
- 問摘要：使用摘要工具。
- 問翻譯：使用翻譯工具。
- 問逐字稿：使用逐字稿工具。

法律資料 MCP 原則：

- 只要問題需要法律資料，可嘗試查詢。
- 查不到就回覆查不到。
- 不應因查不到而生成不存在的裁判或見解。
- 檢索結果應標示來源與日期。

知識庫品質：

- 程序性文書不應進入高信任見解庫。
- 抓不到實質見解者要標記。
- 提示詞外洩內容要排除。
- 重複向量可去重，但正式執行前要有快照或回滾方案。

## 14. 模型日夜切換

私有版有日夜模型切換邏輯。原則：

- 日間以輕量模型處理一般任務。
- 夜間可啟用較大模型處理摘要、檢索整理與深度任務。
- 若日間主模型不符合預期，應檢查排程、模型 sidecar 與 NERV。
- 三模型未上線時，NERV 與健康頁應明確顯示。

檢查：

```bash
curl http://127.0.0.1:5002/health
curl http://127.0.0.1:8090/health
```

若模型離線：

1. 確認模型服務程序。
2. 查看記憶體壓力。
3. 檢查日夜切換排程。
4. 重啟 sidecar。
5. 跑工具調用 live test。

## 15. NAS、檔案預覽、下載與分享

NAS 原則：

- 掛載名稱必須正確，不可掛成 `homes-1` 或 `lumi-1` 之類會破壞 OSC 路徑的名稱。
- NAS 不在區網或重啟中時，不執行批次搬檔。
- 高負載時先確認是否仍可運作，只有持續卡死才考慮重啟。

檔案操作應支援：

- 個別開啟。
- 預覽。
- 下載。
- 分享連結。
- 複製對外文字。

若出現 `Resource deadlock avoided`：

1. 不要重複快速點擊。
2. 檢查檔案是否被 Word、PDF 編輯器、Finder、同步程式鎖住。
3. 由 MAGI 使用安全讀取與重試機制。
4. 必要時先複製到安全暫存再預覽。
5. 若是 NAS 鎖定，等同步完成再重試。

若需要 rename 但資料夾被開啟：

- 先關閉 Finder 視窗。
- 先儲存並關閉 Word 或 PDF 編輯器。
- 若未知編輯器占用，MAGI 可提示使用者，必要時由使用者確認後關閉。
- 不應未經確認強制關閉可能含未存檔內容的文件。

## 16. 通知分流

通知頻道應依性質分開：

- 系統通知：健康、排程、自我修復、磁碟、NAS。
- 法扶一般：巡檢、缺檔、資料補填。
- 法扶派案：新派案、開辦、派案附件。
- 進度回報：逾期案件與 60 天冷卻提醒。
- 檔案通知：下載、歸檔、筆錄與閱卷。
- 錯誤通知：需人工介入。

不得發錯：

- 法扶健康檢查不應跑到法扶二階段或 DC 業務頻道。
- 一般巡檢不應一直發 DC。
- 進度回報才送到需要立即處理的 DC 頻道。
- 筆錄通知只列出有下載的案件。
- 股票或情報報告不應被截斷。

通知內容過長時，應改為：

- 摘要加連結。
- 完整報告存檔。
- 可複製文字輸出。
- 不截斷關鍵內容。

## 17. NERV、排程與自我修復

NERV 是私有版狀態頁。每日應看：

- 主狀態。
- DB。
- 推論服務。
- OCR。
- 向量資料庫。
- NAS。
- 日常稽核。
- 維運健康。

若維運健康顯示 stale，例如某工作超過預期時間未執行：

1. 查看該 job 最近 log。
2. 手動執行一次。
3. 確認排程已 seed。
4. 更新健康狀態。

自我修復提案原則：

- dry run 可自動產生建議。
- execute 模式需有快照、回滾與審批。
- 小型降噪可由維運者手動處理。
- 大型資料刪除或重建需保留確認碼。

## 18. 備份、清理與磁碟控管

備份：

- 私有 DB 故障時，以本機備份為準。
- 每日備份失敗要明確通知。
- DB 雙向同步若原 DB 已故障，不應再報成同一種錯誤，應改成單機備份模式。

清理：

- 可清理可重建快取。
- 可清理過期 runtime report。
- 可清理退役模型快取。
- 可清理測試暫存。
- 不可清理 Paperclip 單機版需要的 JSON、SQLite、pickle、db 或使用者資料。
- 不可清理案件資料、OAuth token、`.env`、DB dump。

磁碟低水位時：

1. 先列出最大目錄。
2. 區分 MAGI 可清理與使用者資料。
3. 優先清理退役服務快取。
4. 再清理可重建下載與 OCR 暫存。
5. 清理後跑健康檢查。

## 19. Git、公開版與脫敏

私有版提交前：

```bash
git status --short
python3 scripts/public_release_audit.py --strict
```

公開版推送前：

```bash
python3 scripts/public_release_audit.py --public-isolation --strict
```

公開版必須移除：

- 私人 NAS 路徑。
- 私人信箱。
- 私人 webhook。
- 私人 token。
- 私有實務見解來源。
- 真實案件資料。
- runtime 報告。
- DB dump。
- portal 截圖。

公開版應保留：

- 安裝精靈。
- 健康檢查。
- 商用上線檢核。
- 通用案件管理。
- 通用檔案、OCR、命名、書狀、行事曆與待辦流程。
- 可由客戶自行填入的 `.env.example`。

若工作樹有未提交檔案，先判斷：

- 程式碼與文件更新：測試後提交。
- runtime/cache/本機資料：不提交。
- 不確定者：先不要 stage。

## 20. 常見狀況處理

### 20.1 MAGI 主頁打不開

```bash
cd ~/Desktop/MAGI_v2
./scripts/magi_cli.sh status
./scripts/magi_cli.sh restart
```

再開 `/health`。若本機可開、外網不可開，檢查 Tailscale/Funnel。

### 20.2 26B 或大模型離線

檢查 sidecar、記憶體、日夜排程與模型健康路由。若記憶體壓力過高，先不要壓力測試。

### 20.3 法扶新派案重複建立

用法扶案號、OSC 案號、當事人、案由與資料夾交叉比對。若已有案件，更新既有案件，不建立新案件。

### 20.4 法扶附件下載了但資料夾看不到

檢查實際下載路徑、資料夾搬移狀態、索引是否更新、NAS 是否同步完成。若 DB 路徑舊了，要更新 DB。

### 20.5 已結案案件打不開

檢查已結案資料夾路徑是否已回寫 DB，並確認網頁版使用同一個開檔 API。

### 20.6 通知重複或送錯頻道

檢查通知分類、冷卻記錄與 webhook 對應。三十天或六十天冷卻不應失效。

### 20.7 報告被截斷

檢查通知長度限制。長報告要存完整檔案，通知只送摘要與連結。

### 20.8 CI 失敗

先看 GitHub annotations，再本機重跑 smoke。不要在 CI 紅燈時宣稱公開版可交付。
